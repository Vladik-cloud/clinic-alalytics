# Архитектурные решения и что бы сделал иначе при большем времени

## Контекст

ТЗ требует веб-отчёт по дампу ClinicIQ: выручка врачей и перенаправляемость, фильтр по периоду, без изменения исходных данных, Docker-запуск, тесты и деплой на VPS.

## Принятые решения

### 1. Стек: FastAPI + PostgreSQL + React

- **FastAPI** — быстрый REST, типизация, удобные тесты через `TestClient`.
- **PostgreSQL** — типичный выбор для дампов MIS/ClinicIQ; аналитика на SQL прозрачна.
- **React + Recharts** — один экран отчёта с bar-chart и таблицами, без тяжёлого BI.

### 2. Read-only доступ к данным

- Роль `cliniciq_readonly` с `GRANT SELECT`.
- Соединение API с `default_transaction_read_only=on`.
- Нет INSERT/UPDATE/DELETE в коде приложения.

Так выполняется требование «данные в БД не менять» на уровне СУБД, а не только дисциплины разработчика.

### 3. Инициализация БД

Скрипт `db/init/00_init.sh`:

1. Если есть валидный `data/dump.sql.gz` — импорт дампа.
2. Иначе — демо-схема и seed для проверки без живой ссылки.
3. Всегда — выдача прав read-only.

### 4. Адаптер схемы (`schema_map.py`)

Реальный дамп ClinicIQ может отличаться от демо (`employees`, `appointments`, `invoice_items`). Реализованы:

- эвристический поиск таблиц/колонок по `information_schema`;
- переопределение через env (`SCHEMA_*`).

Аналитические запросы строятся динамически, но логика метрик фиксирована в `analytics.py`.

### 5. Определение перенаправляемости

```
Для каждого пациента: first_visit = MIN(visit_at)
Врач first_visit.doctor_id — «первичный»
Перенаправление = ∃ визит позже с doctor_id ≠ первичного
```

Показатель по врачу: `перенаправленные / все с первичным приёмом у этого врача × 100%`.

Это прямое чтение формулировки ТЗ. Не учитываются: отмены, no-show (фильтр `status = completed` где колонка есть), направления внутри одного дня.

### 6. Docker Compose

Три сервиса: `db`, `api`, `web` (nginx проксирует `/api` на backend).  
`make up` = `docker compose up --build -d`

## Что бы улучшить при большем времени

1. **Проверка на реальном дампе** — сейчас ссылка expired; после получения свежего дампа — зафиксировать точный `schema_map` в конфиге и добавить snapshot-тесты SQL.
2. **Materialized views** — для больших объёмов предрасчёт `revenue_by_doctor_mv` и `referral_mv` по ночам, API только читает витрины.
3. **Кэш отчёта** — Redis с ключом `(start, end)` и TTL 5–15 мин.
4. **Аутентификация** — basic auth или OAuth2 перед публикацией на VPS.
5. **CI** — GitHub Actions: `pytest`, `docker compose up`, smoke `curl /api/health`.
6. **Уточнение бизнес-правил** — учитывать ли перенаправление к тому же врачу в другом филиале, вес услуг в выручке (оплата vs начисление), НДС.
7. **i18n и доступность** — aria-labels, экспорт PDF/Excel.
8. **Observability** — structured logs, Prometheus metrics на latency запросов отчёта.
